Specter Help

Specter is a Ghidra extension by Cold Entry LLC for agent-assisted reverse engineering.

Specter can inspect the active program using decompilation, disassembly, cross references, and regex-based searches. It can also update the active program database by renaming listing items, adding listing or decompiler-visible comments, renaming parameters and local variables, and creating functions at undefined locations.

The terminal starts in DSL mode with the specter dsl> prompt, where entered text runs as Specter DSL or SQL-like statements against the active program. Press Ctrl+M to switch to chat mode with the specter chat> prompt, where entered text is sent to the configured chat model. Slash commands work in both modes, including /model for model profiles, /subagent for saved sub-agent definitions, /resetprefs to clear Specter preferences after confirmation, /new to clear the current conversation, /thinking to control thinking visibility, and /dsl to show DSL help. Up and Down recall input history for the current mode. DSL and chat histories are saved across Ghidra sessions and cleared by /resetprefs.

The assistant uses a RunSpecterDslQuery tool for most read-only program context, including function queries, strings, symbols, defined data, references, decompilation, instruction listings, grouping, ordering, and DSL loops. Separate tools remain available for current cursor location, datatype work, sub-agents, and approval-gated database edits.

Use /model to choose the model provider for each saved profile. Select OpenAI-compatible for the official OpenAI API and compatible local servers such as llama.cpp. For llama.cpp, use the server's /v1 base URL, for example http://127.0.0.1:8080/v1, leave the API key blank unless the server requires one, and enter the model name or alias accepted by the server. Select Ollama native for Ollama's native /api/chat endpoint.

DSL mode accepts plain SQL over the FUNCTION, STRING, DATA, REFERENCE, and SYMBOL virtual tables. Add UNIQUE or DISTINCT after SELECT to keep only the first row for each selected value tuple. Example: SELECT name, entry FROM FUNCTION WHERE name LIKE 'FUN_%' ORDER BY entry LIMIT 20; Hexadecimal numeric literals are supported, for example SELECT entry, name FROM FUNCTION WHERE entry >= 0x4044d1; The BETWEEN, IN, NOT IN, LIKE, and NOT LIKE operators are supported, and predicates may be combined with AND, OR, and parentheses.

SQL JOIN supports explicit inner joins with table aliases, ON predicates, and qualified columns such as caller.name. To find callers, join REFERENCE.from_address against a function body range because the reference source is the callsite address: SELECT UNIQUE caller.entry, caller.name, r.from_address FROM REFERENCE r JOIN FUNCTION callee ON r.to_address = callee.entry JOIN FUNCTION caller ON r.from_address >= caller.body_min AND r.from_address <= caller.body_max WHERE callee.entry = 0x4044d1 AND r.type IN ('COMPUTED_CALL', 'UNCONDITIONAL_CALL') ORDER BY caller.name LIMIT 100;

The STRING table exposes defined string data with address, value, data_type, and length columns. Example: SELECT * FROM STRING WHERE value LIKE 'usage:%'; Another example: SELECT address, value, length FROM STRING WHERE value LIKE '%http%' ORDER BY address LIMIT 20;

The DATA table exposes defined data items for constants and globals with columns including address, name, data_type, value, value_text, is_constant, is_global, and reference_count. Examples: SELECT address, name, data_type, value_text FROM DATA WHERE is_global = true ORDER BY address LIMIT 20; SELECT address, data_type, value FROM DATA WHERE is_constant = true ORDER BY address LIMIT 20;

The REFERENCE table exposes raw Ghidra references with from_address, to_address, type, and source columns. Examples: SELECT from_address, to_address, type FROM REFERENCE WHERE to_address = 0x4044d1 LIMIT 20; SELECT from_address, to_address, type FROM REFERENCE WHERE type IN ('COMPUTED_CALL', 'UNCONDITIONAL_CALL') LIMIT 50;

The SYMBOL table exposes primary symbols with address, name, namespace, and source columns.

SELECT UNIQUE removes duplicate result rows before applying LIMIT. Example: SELECT UNIQUE namespace FROM FUNCTION ORDER BY namespace LIMIT 20;

The first DSL construct is a read-only loop: FOR level IN SELECT callgraph_level(entry) FROM FUNCTION ORDER BY callgraph_level(entry) DESC LIMIT 2; DO SELECT entry, name FROM FUNCTION WHERE callgraph_level(entry) = :level LIMIT 5; END;

Write-capable SQL statements support listing renames: UPDATE FUNCTION SET name = 'parse_header' WHERE entry = 0x4044d1; UPDATE FUNCTION SET name = prompt('Suggest a function name') WHERE entry = 0x4044d1; UPDATE DATA SET name = 'global_config' WHERE address = 0x405000; UPDATE SYMBOL SET name = 'dispatch_table' WHERE address = 0x406000; UPDATE supports FUNCTION.name, DATA.name, and SYMBOL.name, requires a WHERE clause, and records renames with Ghidra SourceType.AI.

The SQL built-in callgraph_level(entry) computes an acyclic callgraph level using Ghidra's AcyclicCallGraphBuilder. Root and entry-point functions are level 0, and callees increase in level. Example: SELECT entry, name, callgraph_level(entry) FROM FUNCTION ORDER BY callgraph_level(entry) DESC LIMIT 50;

Select expressions may use AS alias or a bare alias. The aggregate built-ins count(*), count(1), and count(column) count matched rows. Example: SELECT count(1) func_count FROM FUNCTION;

GROUP BY supports selected aliases and expressions. Grouped output is sorted by group key by default, with NULL groups last. Example: SELECT callgraph_level(entry) lvl, count(*) FROM FUNCTION GROUP BY lvl;

The SQL built-in concat(value1, value2, ...) joins two or more values as text and treats NULL arguments as empty strings. Example: SELECT concat(namespace, '::', name), entry FROM FUNCTION LIMIT 20;

The SQL built-ins decompilation(entry) and instructions(entry) are supported in the select list and render multiline row blocks. Examples: SELECT entry, name, decompilation(entry) FROM FUNCTION WHERE entry = 0x4044d1; SELECT entry, name, instructions(entry) FROM FUNCTION WHERE entry = 0x4044d1;

The SQL built-in prompt(expression) invokes the configured model from the select list and renders the response as a multiline row block. Constant prompts ignore the selected table's row count and return one row: SELECT prompt("hello world"); SELECT prompt("hello world") FROM FUNCTION; Row-dependent prompts invoke the model for each displayed row: SELECT name, prompt(name) FROM FUNCTION LIMIT 2; During prompt() evaluation, the model can use read-only inspection tools and read-only RunSpecterDslQuery calls for additional program context, but cannot run database-editing tools or nested prompt() calls.

Specter seeds several sub-agents by default: Explore for surrounding function and callgraph context, SummarizeFunction for focused single-function summaries, StringsAndXrefs for string-led triage, Types for datatype and structure analysis, Naming for rename suggestions, and Callgraph for caller and callee mapping.

Configured sub-agents are available to the main assistant as tools. The assistant can list the saved sub-agents, inspect one sub-agent's saved instructions, and invoke one by name to delegate a focused task using the current model and the active program context.

These edits are intended for operator-directed annotation work inside the current program. As the extension grows, this page should document those workflows and any trust or execution boundaries enforced inside Ghidra.
