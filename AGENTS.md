# Repository Guidelines

## Project Structure & Module Organization

Specter is a Java Ghidra extension. Core plugin code lives in `src/main/java/com/coldentry/specter/`, including UI classes, chat backends, Ghidra tool adapters, and the SQL/DSL implementation. Ghidra help content is in `src/main/help/help/topics/specter/help.html` with the TOC at `src/main/help/help/TOC_Source.xml`. Static resources belong in `src/main/resources/`. Extension metadata and build configuration are at the repository root: `extension.properties`, `Module.manifest`, and `build.gradle`. Generated artifacts are written to `build/` and `dist/`; do not edit them manually.

## Build, Test, and Development Commands

Set `GHIDRA_INSTALL_DIR` before building:

```bash
export GHIDRA_INSTALL_DIR=/absolute/path/to/ghidra
./gradlew buildExtension
```

`./gradlew buildExtension` compiles Java, indexes help, builds the jar, and creates the extension zip under `dist/`. Use `./gradlew compileJava` for a faster compile-only check. `./gradlew distributeExtension` invokes Ghidra's extension distribution task.

## Coding Style & Naming Conventions

Use Java with tabs for indentation, matching the existing source. Keep classes package-private unless Ghidra requires public visibility. Follow the existing `Specter*` naming pattern for plugin components and keep single-purpose service classes, for example `SpecterSqlQueryService` and `SpecterDslService`. Prefer clear error messages from parser and tool code because they surface directly in the terminal. Avoid unrelated formatting churn.

## Testing Guidelines

There is no dedicated automated test suite yet. At minimum, run:

```bash
./gradlew buildExtension
```

For parser or DSL changes, manually exercise representative `/dsl` commands in Ghidra, including failure cases. For write-capable changes, verify Ghidra transactions roll back on errors and that edits use `SourceType.AI` where applicable.

## Commit & Pull Request Guidelines

Recent commits use short imperative messages, such as `Add SQL GROUP BY support` or `Add SQL update and boolean predicates`. Keep commits focused and include docs/help updates with user-facing behavior changes. Pull requests should describe the behavior change, list verification performed, and note any Ghidra/manual testing. Include screenshots only for UI-visible changes.

## Security & Configuration Tips

Do not commit API keys, local model endpoints, or user preference data. Model profiles and sub-agent definitions are stored in Java preferences, not source. Keep third-party dependencies declared in `build.gradle`; built copies in `lib/` are generated by the extension build.
